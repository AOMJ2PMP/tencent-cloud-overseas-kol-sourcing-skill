# 腾讯云海外达人筛选

这个 skill 的目标不是列出“看起来很有名”的账号，而是产出在指定市场、预算、平台、时间和内容形态下，真实有机会与腾讯云达成合作并交付有效内容的达人组合。

默认使用中文写内部分析；达人对外 brief、选题和 outreach 文案按目标市场使用英语或用户指定语言。

## 不可跳过的五道门

1. **Brief gate**：硬性条件未明确时，先追问，不交付最终名单。
2. **Product truth gate**：必须调研腾讯云产品、目标用户、市场可用性、可演示路径和 claim 风险。
3. **Creator evidence gate**：每位主推达人必须有近期内容、受众/地域、表现和商业合作证据。
4. **Budget gate**：报价区间与推荐组合必须落在实际预算内，并预留执行费用。
5. **Collaboration gate**：必须判断达人是否可联系、愿意接商业合作、内容是否自然、是否存在明显冲突或档期风险。

若某一道门证据不足，明确写 `待验证`，不要把推测写成事实。

## 工作流

### 1. 读取 brief，建立硬条件

先读取 [references/brief-intake.md](references/brief-intake.md)。

至少确认：

- 腾讯云产品名称及官方页面、文档或 GitHub 仓库
- campaign 目标和主要 KPI
- 目标国家/地区；“达人必须 base 当地”还是“受众在当地”
- 平台、内容语言、内容形态和交付数量
- 总预算、币种、预算是否含税/差旅/授权/投流/代理费
- 预计上线时间、可拍摄或可体验条件
- 达人数量、赛道、必须项和排除项

如缺少会改变筛选结果的信息：

1. 每轮只问最影响结果的 2-3 个问题。
2. 先问硬条件，再问偏好；必要时进行第二轮。
3. 可在等待回复时做只读产品调研，但不得提前给出“最终推荐”。
4. 用户明确要求直接推进时，列出假设；硬条件仍不能凭空猜测。

### 2. 强制完成腾讯云产品调研

读取 [references/product-research.md](references/product-research.md)，优先浏览当前官方来源：

- 腾讯云产品页、国际站文档、API/SDK 文档
- GitHub 仓库、README、release、issue 和示例
- 定价、地域可用性、合规/安全说明
- 官方案例、发布公告、合作伙伴或活动资料
- 必要时用可信第三方来源验证市场语境和竞品

输出一张 `Product Truth Card`，至少回答：

- 产品到底解决什么问题，给谁用
- 在目标市场是否已可用，体验前置条件是什么
- 达人能在内容里真正演示什么
- 观众为什么会关心
- 哪些能力、数字、安全或性能表述需要产品同学确认
- 哪些达人内容类型最自然，哪些会显得强行植入
- 腾讯云为什么值得这个达人合作，而不只是“给钱背书”

没有 Product Truth Card，不进入最终推荐。

### 3. 建立候选池

围绕产品的用户问题、使用场景、竞品类别和内容机制搜索，不只搜产品名。候选来源包括：

- TikHub 的 YouTube/X 搜索、账号资料和近期内容
- 平台原生搜索、Google/网页搜索
- 产品社区、GitHub、会议演讲者、newsletter、播客和本地媒体
- 已知同类赞助内容中的创作者

先建立 2-4 倍于交付数量的长名单，再验证和淘汰。严格地域要求时，主名单不得混入“可以飞过去”但并非当地 base 的全球达人；这类账号只能放在单独的 `Wildcard` 区。

### 4. 用 TikHub 统一验证

读取 [references/tikhub-data-map.md](references/tikhub-data-map.md)。

每位候选尽量使用相同时间窗和样本量：

- Profile：粉丝数、简介、公开地点、账号链接
- Recent content：YouTube 长视频通常取最近 10-20 条；X 通常取最近 30-50 条
- Performance：中位播放/曝光、互动中位数、互动率、发布频率、异常爆款
- Relevance：近 12 个月至少 3 条与产品/场景相关的内容，或说明为什么样本不足
- Commercial signals：近期赞助、商务邮箱/经纪、media kit、付费合作披露

遵守这些原则：

- 用中位数而不是均值作为基础表现。
- Shorts、长视频、直播和 X 帖子分开比较。
- 赞助内容与自然内容分开看。
- 同一批候选使用相同样本量和时间窗。
- TikHub 调用前检查环境变量、预计调用量和积分；分页必须设上限。
- API key 只从 `TIKHUB_API_KEY` 读取，绝不写入文件或提交。

### 5. 验证地域、语言和受众

`base 当地` 与 `受众在当地` 是两个不同字段：

- 严格当地 base：至少需要两个相互独立的公开证据，例如账号自述地点 + 官网/公司信息/当地长期内容/公开活动资料。
- 受众地域：优先使用 creator media kit 或平台 analytics；没有数据时写 `未验证`，不能用达人所在地代替。
- 语言：以近期主要内容语言为准，不只看简介。

只要用户说“严格 base”，证据不足的候选就不能进入主推名单。

### 6. 判断与腾讯云合作的现实性

读取 [references/creator-evaluation.md](references/creator-evaluation.md)。重点回答：

- 他的受众是否真的会用或影响购买这个产品
- 他的自然内容机制能否承载 demo、教程、探店、访谈或案例
- 过去 12 个月是否接过相近的 B2B、开发者、云或科技商业合作
- 是否有公开商务联系方式、经纪或合作入口
- 是否存在仍有效的竞品排他、明显的品牌冲突或交付障碍
- 在当前预算和时间内，什么 scope 最可能谈成
- 达人为什么愿意做：内容价值、独家体验、技术新意、当地场景、受众需求或商业回报

不得因为达人公开讨论某类敏感议题就推断其私人属性或政治立场。只记录与本次商业合作直接相关、可引用的公开品牌安全事实。

### 7. 预测报价并生成预算组合

读取 [references/pricing-and-budget.md](references/pricing-and-budget.md)。

报价必须输出 `low / base / high` 区间和置信度。至少考虑：

- 近期可比内容的中位播放/曝光，不以粉丝数单独定价
- dedicated、integration、thread、short-form 等交付形式
- 制作复杂度、差旅、产品体验和技术准备
- 使用权、白名单投流、二次剪辑、排他、加急和修改轮次
- 税费、代理费和汇率

默认预留总预算的 10%-20% 给执行项。主推组合的 base 预测应落在可用达人预算内；只有 low 能勉强落入时标记 `可谈但有风险`，high 仍能落入时标记 `稳妥`。

可使用：

```bash
node scripts/score_candidates.mjs examples/sample-candidates.json
```

脚本只帮助做一致的评分和组合，不替代人工证据判断。

### 8. 评分、淘汰与组合

先执行硬性淘汰，再评分。硬性淘汰包括：

- 不符合必选平台、语言或严格地域要求
- 明显不符合内容赛道或无法自然演示产品
- 已知基础报价超过整个可用预算
- 明确的品牌安全排除项、有效排他或无法赶上档期
- 无法找到足够证据确认账号真实性和近期活动

评分权重：

| 维度 | 权重 |
| --- | ---: |
| 产品与内容自然度 | 25 |
| 受众与目标市场匹配 | 20 |
| 近期表现质量 | 15 |
| 商业合作可行性 | 15 |
| 预算匹配 | 15 |
| 品牌安全与交付风险 | 10 |

主推通常要求总分不低于 70/100，且产品内容自然度、合作可行性都不低于 3/5。证据置信度低的高分候选只能放在备选。

组合不要只追求总粉丝数。优先让不同角色互补，例如：

- 1 位有声量的品类/当地头部 + 1-3 位高匹配教程型达人
- 1 位开发者权威 + 1 位擅长把技术讲给更广人群的 creator
- 1 位本地生活场景达人 + 1 位科技解释型达人

### 9. 交付可执行结果

读取 [references/output-contract.md](references/output-contract.md)，最终至少包含：

1. 一页决策摘要
2. Brief 已确认项、假设和未确认项
3. Product Truth Card
4. 主推候选表与备选表
5. 2-4 个预算组合及其取舍
6. 报价模型、预算预留和置信度
7. 每位候选的推荐选题/内容机制
8. 风险、待验证项和联系优先级
9. Evidence ledger：来源链接、数据日期、样本量、TikHub 与其他来源的字段边界
10. 明确列出被淘汰的知名达人及理由，防止名单回摆

## 最终质量检查

在交付前逐项确认：

- [ ] Brief 的硬条件已经确认或明确标注假设
- [ ] 已浏览当前腾讯云官方产品资料
- [ ] 产品市场可用性和可演示路径已确认
- [ ] 主推达人全部符合硬性地域与语言要求
- [ ] 近期表现使用统一样本和中位数
- [ ] 每位主推都有合作可行性证据
- [ ] 报价不是只由粉丝数推导
- [ ] 推荐组合的 base 成本在预算内
- [ ] 已留执行预算，未把全部钱都分给达人 fee
- [ ] 事实、推测、待产品确认项已分开
- [ ] 所有关键数字带来源和抓取日期
- [ ] 没有为了“看起来头部”塞入不现实的大牌

任一关键项失败时，不要声称“名单已完成”。
