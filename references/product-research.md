# Product Research

产品调研是达人筛选的前置条件，不是名单完成后的补充说明。

## 来源优先级

1. 腾讯云官方产品页、国际站文档、API/SDK 文档、定价和地域列表
2. 腾讯云 GitHub 仓库、README、release、issue、示例和 license
3. 官方公告、客户案例、合作伙伴公告、活动资料
4. 产品团队提供的 brief、FAQ、demo 和 claim 清单
5. 可信第三方评测、行业报告和竞品资料
6. 社媒讨论，只用于发现问题和内容角度，不单独作为产品事实

所有可能变化的信息都要在本次任务中重新浏览，不能只依赖记忆。关键 claim 紧邻引用来源和访问日期。

## Product Truth Card

| 字段 | 要回答的问题 |
| --- | --- |
| One-line truth | 用普通用户能懂的一句话解释产品 |
| Product stage | GA、preview、开源项目、活动 demo，还是概念验证 |
| Target users | 谁实际安装、购买、部署或影响采购 |
| Primary problem | 用户当前为什么痛，替代方案是什么 |
| Market readiness | 目标区域是否可用，数据驻留/付款/账号是否有限制 |
| Demo path | 达人可以在镜头或屏幕上完成的 3-6 个动作 |
| Time to value | 从开始到看到结果要多久，需要什么环境 |
| Differentiation | 能被证据支持的差异，不写空泛“领先” |
| Proof | benchmark、客户案例、stars、adoption 或官方说明 |
| Claim risks | 性能、安全、隐私、合规、数字和比较性表述 |
| Audience value | 即使不购买，观众能学到什么 |
| Creator value | 为什么这个选题值得达人做 |
| Best formats | 教程、评测、访谈、探店、案例、直播等 |
| Bad formats | 会显得强行植入或无法验证的形式 |

## 按产品类型补充

### 开发者或数据库产品

- 实际安装和运行官方 quickstart。
- 检查依赖、支持语言、数据库、云账号和系统要求。
- 找到最小可复现 demo 和失败路径。
- 对 benchmark 写清测试条件与来源。
- 判断教程达人能否独立复现，而不是只念 README。

### SaaS/API 产品

- 确认可注册国家、免费额度、试用、价格和 API access。
- 记录需要信用卡、企业认证或白名单的步骤。
- 为达人准备可演示的输入、输出和对照组。

### 本地线下或硬件体验

- 确认门店、开放时间、拍摄许可、体验资格和工作人员配合。
- 确认产品在当地是真实上线还是活动限定。
- 把隐私、安全和用户同意设计进内容脚本。

## Claim 分类

- `Verified`：本次从官方或可复现来源确认。
- `Product confirmation needed`：官方有表述，但范围/条件不清。
- `Creator test`：应由达人独立测试并说明环境。
- `Do not claim`：无来源、比较性过强或会引发误导。

## 进入达人搜索前的最低标准

- 能准确解释产品和用户。
- 能确认目标市场可用性或明确未知项。
- 至少有一个可拍/可录屏的真实 demo。
- 知道主要 claim 风险。
- 能写出 3 个以受众价值为中心的内容角度。
